EdgeNet is a distributed edge cloud, in the family of PlanetLab, GENI, JGN-X, and PlanetLab Europe. It is a modern distributed edge cloud, incorporating advances in Cloud technologies over the past few years. EdgeNet is based on industry-standard Cloud software, with Docker as the containerization technology and Kubernetes as the node manager and deployment solution. It is an opt-in global Kubernetes cluster; once a user has authenticated with this portal and been approved, she will be able to use standard Kubernetes tools and technologies to deploy an application across the EdgeNet infrastructure. As a software-only infrastructure, its design goal is that any local virtual machine with Internet connectivity should be able to join the EdgeNet worldwide infrastructure in under five minutes, and any developer should be able to deploy his application across EdgeNet using standard, familiar Cloud technologies within five minutes.

Why EdgeNet?

Open Source

EdgeNet is an open source project to contribute back to the Kubernetes community. The backend, an extension of the Kubernetes API, is an API written in Go that is open to those who have auth. It gets the benefit of client-go and Go packages for talking to the Kubernetes API of EdgeNet and makes use of the Namecheap API and the GeoLite2 database. You are very welcome to contribute our community! Let's visit our GitHub repository:

Kubernetes Based

Kubernetes is a well-known and an open-source system for the orchestration of containerized applications. More explicitly, Kubernetes automates deployments, and it also scales and manages containerized applications across a cluster. Originally, Kubernetes appears itself as a solution for cloud infrastructure, but, there is a strong community who wants to bring it to Edge. And, EdgeNet is one of the players to do so by building itself up on Kubernetes.

Docker Technology

Thanks to containerization technology, it easier to build and deploy applications across servers in a way that is a reliable and scalable way. It also provides us an opportunity to avoid setting up an operating system per application. Briefly, Docker is a platform to build, manage, and secure applications and also to may deploy them. EdgeNet users make use of Docker containers to pack and deploy their applications.

Edge Features

EdgeNet is yet to focus on Device Edge, so the devices, such as smartphones, laptops, and sensors, that really have limited spare compute and data storage capability with connectivity problems. Distinct from infrastructure edge as device edge uses device resources, whereas infrastructure edge uses edge nodes such as individual servers or other sets of computing resources. So, EdgeNet has established a Kubernetes edge cluster that the edge nodes, with cloud-like capabilities are located in distributed locations at infrastructure edge. Moreover, EdgeNet makes it easy for third parties to add, to contribute nodes into EdgeNet cluster in the way Kubernetes users do by using .YAML configuration files. In addition to this, EdgeNet presents Selective Deployment that provides ability of deploying applications based on the node geolocation.

Selective Deployment allows users to deploy containerized applications to a widely distributed set of nodes by choosing the nodes by their locations. Users can do so based on continents, countries, states/regions, and cities, as well as polygons describes using latitudes and longitudes. Please click here for further details.

Multitenancy

EdgeNet inherits the multitenancy model, in use and development, mainly from PlanetLab. A site is a physical location where PlanetLab nodes are located, whereas a site in EdgeNet is not required to provide a node. But, they both own users and slices, additionally, sites in EdgeNet have an added feature called Project inherited from Fed4FIRE.

The central admins of EdgeNet approve the establishment of sites, each site having its own local site admins called a PI, so Principal Investigator. A PI approves the creation of individual user accounts. PIs can delegate user approval, creation of projects and slices to managers. And by giving a tech role can make a user be responsible for nodes that are contributed to the EdgeNet cluster. And a user is anyone who develops and deploys applications on EdgeNet. A slice is a set of allocated resources distributed across PlanetLab, and to most users, a slice means UNIX shell access to certain PlanetLab nodes, whereas a slice means a set of allocated resources on the cluster in EdgeNet. As a final one, Project groups users to give them permission to create slices independently.

Architecture

Resources are endpoints of the Kubernetes API to store objects separately by kinds. Kubernetes has a built-in feature that allows developers to generate customized resources dedicated their particular needs. These solutions called as custom resources in Kubernetes, which enables users to create, edit, and delete objects belong to a custom resource in the way they do for built-in resources by invoking kubectl. Thus, these custom solutions become a part of Kubernetes and thanks to this feature we can avoid forcing user to learn a new syntax. In EdgeNet, we use CustomResourceDefinitions (CRDs) to extend the Kubernetes API. Although CustomResourceDefinitions don't provide flexibility because of its restrictions, they don't require to develop a new API server, saves us from a plent of coding task, as they use the native Kubernetes API Server.

Aimed Architecture

The idea behind the development of EdgeNet is meeting distributed testbed requirements by using a common framework that is Kubernetes in this case. EdgeNet aims being fully aling with Kubernetes to be able to contribute back to the community. The custom resources feature of Kubernetes allows us to do so.
